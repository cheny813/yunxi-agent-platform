# 01. 介绍

## 什么是 AI Agent

### 定义与概念

**AI Agent（人工智能代理）** 是一种能够感知环境、做出决策并执行行动的自主系统。与传统软件不同，Agent 具有以下核心特征：

| 特征 | 说明 | 与传统软件对比 |
|------|------|----------------|
| **自主性** | 能够独立运行，无需人工逐步指令 | 传统软件：按固定流程执行 |
| **反应性** | 能感知环境变化并实时响应 | 传统软件：被动等待输入 |
| **主动性** | 能主动发起行动达成目标 | 传统软件：仅响应用户操作 |
| **推理能力** | 能进行逻辑推理和规划 | 传统软件：基于规则的硬编码逻辑 |
| **学习能力** | 能从经验中学习和改进 | 传统软件：功能固定不变 |

### Agent 的工作原理

```
┌─────────────────────────────────────────┐
│              AI Agent 工作循环            │
├─────────────────────────────────────────┤
│                                         │
│  ┌─────────┐    ┌─────────┐    ┌─────┐ │
│  │ 感知    │───→│ 思考    │───→│ 行动 │ │
│  │ (Perceive)│   │ (Think) │    │(Act)│ │
│  └────┬────┘    └────┬────┘    └──┬──┘ │
│       │              │            │    │
│       └──────────────┴────────────┘    │
│              ↑ 环境反馈 ↓               │
│                                         │
└─────────────────────────────────────────┘
```

**循环过程**：
1. **感知**：通过传感器（API、数据库、用户输入）获取环境信息
2. **思考**：分析信息、推理、规划下一步行动
3. **行动**：执行决策（调用工具、生成回复、修改数据）
4. **反馈**：观察行动结果，进入下一轮循环

### 为什么需要 Agent

**传统软件开发的问题**：
- 需求变化时需要修改代码、重新部署
- 复杂业务逻辑导致代码难以维护
- 无法处理模糊、不确定的场景

**Agent 的优势**：
- 通过自然语言描述需求，无需修改代码
- LLM 处理模糊逻辑，适应不确定性
- 自主决策，减少人工干预

## 什么是多 Agent 系统

### 单 Agent 的局限

单个 Agent 虽然强大，但在复杂场景下存在局限：

| 问题 | 说明 | 示例 |
|------|------|------|
| **知识局限** | 单个 Agent 难以掌握所有领域知识 | 营养专家不懂成本核算 |
| **计算瓶颈** | 复杂任务处理时间过长 | 同时分析营养、成本、合规 |
| **单点故障** | Agent 故障导致整个系统不可用 | 营养 Agent 崩溃 |
| **难以扩展** | 新增能力需要修改原有 Agent | 增加口味评估功能 |

### 多 Agent 系统的优势

**多 Agent 系统（Multi-Agent System, MAS）** 将复杂任务分解给多个专业 Agent：

```
用户请求："分析这份食谱"

┌─────────────────────────────────────────┐
│           Supervisor Agent              │
│           （任务协调者）                  │
└──────────────┬──────────────────────────┘
               │
    ┌──────────┼──────────┬──────────┐
    ▼          ▼          ▼          ▼
┌───────┐  ┌───────┐  ┌───────┐  ┌───────┐
│营养分析│  │成本计算│  │合规检查│  │口味评估│
│ Agent │  │ Agent │  │ Agent │  │ Agent │
└───┬───┘  └───┬───┘  └───┬───┘  └───┬───┘
    │          │          │          │
    └──────────┴──────────┴──────────┘
               │
               ▼
        ┌─────────────┐
        │   结果聚合    │
        │  完整分析报告 │
        └─────────────┘
```

**优势**：
- **专业分工**：每个 Agent 专注特定领域
- **并行处理**：多个 Agent 同时工作，提高效率
- **容错性**：单个 Agent 故障不影响整体
- **可扩展性**：新增 Agent 无需修改现有系统

### 多 Agent 协作模式

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| **Supervisor 模式** | 一个主管 Agent 协调多个专家 Agent | 复杂任务分解 |
| **流水线模式** | Agent 按顺序处理，前一个输出作为后一个输入 | 数据处理流程 |
| **投票模式** | 多个 Agent 独立决策，最终投票决定 | 需要高可靠性的决策 |
| **竞争模式** | 多个 Agent 竞争执行，选择最优结果 | 创意生成、方案设计 |

## 为什么需要 yunxi Agent Platform

### 企业级 AI 应用的挑战

将多 Agent 系统应用于企业环境，面临以下挑战：

**1. 多平台接入挑战**

企业用户分散在不同平台：
- 内部员工使用企业微信
- 供应商使用钉钉
- 客户使用 Web 或 APP

**问题**：需要为每个平台开发适配层，代码重复，维护困难。

**2. 业务隔离挑战**

企业有多个业务领域：
- 营养配餐
- 财务审计
- 人力资源

**问题**：不同领域的数据、规则、权限需要隔离，避免相互干扰。

**3. 规则管控挑战**

企业需要合规和审计：
- 数据访问权限控制
- 操作审计日志
- 敏感信息过滤

**问题**：在每个 Agent 中重复实现规则检查，代码冗余。

**4. 生产运维挑战**

企业级应用需要：
- 高可用性（99.9%+）
- 性能监控
- 故障排查

**问题**：基础框架缺乏企业级运维能力。

### 现有框架的能力边界

**AgentScope-Java**：优秀的基础 Agent 编程框架
- ✅ ReAct 推理模式
- ✅ MCP 工具调用
- ✅ A2A 多 Agent 协作

**但在企业级应用中**：
- ❌ 需要自行实现平台适配
- ❌ 缺乏业务隔离机制
- ❌ 没有内置规则引擎
- ❌ 缺少运维基础设施

### yunxi Agent Platform 的定位

**不是替代，而是封装和扩展**

```
┌─────────────────────────────────────────────────────────────┐
│  应用层：yunxi Agent Platform                                  │
│  - 统一网关：多平台接入、认证、限流                             │
│  - 业务编排：Supervisor 模式、场景路由                         │
│  - 规则引擎：PRE/RUNTIME/POST 三阶段规则                       │
│  - 智能服务：自适应策略、自我改进、监控                         │
├─────────────────────────────────────────────────────────────┤
│  框架层：AgentScope-Java + yunxi Core                        │
│  - ChatAppService：对话编排与 Agent 推理执行                   │
│  - MCP Client：工具调用                                      │
│  - A2A Protocol：Agent 间通信                                │
├─────────────────────────────────────────────────────────────┤
│  模型层：DashScope / OpenAI / Claude                          │
│  - 大语言模型能力                                              │
└─────────────────────────────────────────────────────────────┘
```

**核心价值**：
- **降低门槛**：通过配置而非代码创建复杂 Agent 系统
- **企业特性**：内置多平台适配、规则管控、运维监控
- **可扩展性**：SPI 机制支持业务定制，不侵入核心代码

## 核心理论基础

### 1. ReAct 推理模式

**理论来源**：ReAct (Reasoning + Acting) 是普林斯顿大学提出的 Agent 推理范式。

**核心思想**：将推理（Reasoning）和行动（Acting）结合，通过交替进行思考和行动来解决问题。

```
ReAct 循环：

用户问题 → 思考(Thought-1) → 行动(Action-1) → 观察(Observation-1)
                                              ↓
                    ← 思考(Thought-2) ← 行动(Action-2) ← 观察(Observation-2)
                                              ↓
                                         ...
                                              ↓
                                         最终答案
```

**示例**：
```
问题：北京今天天气如何？

思考-1：我需要查询北京的天气信息。
行动-1：调用天气查询工具，参数：location="北京"
观察-1：{ "temperature": "25°C", "weather": "晴" }

思考-2：我已经获取到天气信息，可以回答用户了。
最终答案：北京今天晴天，温度25°C。
```

### 2. MCP 协议

**理论来源**：MCP (Model Context Protocol) 是 Anthropic 推出的开放标准。

**核心思想**：标准化 LLM 与外部工具的通信协议，让工具像 USB 设备一样即插即用。

**协议层次**：
```
┌─────────────────────────────────────────┐
│  应用层：工具定义 (Tool Definition)        │
│  - 工具名称、描述、参数 Schema             │
├─────────────────────────────────────────┤
│  协议层：JSON-RPC 2.0                     │
│  - 请求/响应格式                          │
│  - 错误处理                               │
├─────────────────────────────────────────┤
│  传输层：HTTP / SSE                       │
│  - 实时通信                               │
└─────────────────────────────────────────┘
```

**优势**：
- **标准化**：任何支持 MCP 的 Agent 都能调用任何 MCP 工具
- **安全性**：工具可以独立部署，通过认证控制访问
- **可扩展性**：新增工具无需修改 Agent 代码

### 3. 领域驱动设计 (DDD)

**理论来源**：Eric Evans 的《领域驱动设计》

**核心概念**：

| 概念 | 说明 | 在本框架中的体现 |
|------|------|------------------|
| **领域 (Domain)** | 业务问题的范围 | 营养域、财务域、运维域 |
| **限界上下文** | 领域的边界 | 通过 DomainContributor 定义 |
| **实体 (Entity)** | 有唯一标识的对象 | Agent、Scene、Rule |
| **值对象** | 无标识的属性集合 | AgentConfigDto、ChatRequest |
| **领域服务** | 跨实体的业务逻辑 | ConversationDomainService、AgentDomainService |

**为什么使用 DDD**：
- 业务逻辑与技术实现分离
- 便于业务专家与开发人员沟通
- 支持复杂业务系统的演进

### 4. 依赖倒置原则 (DIP)

**理论来源**：Robert C. Martin 的 SOLID 原则

**核心思想**：
- 高层模块不应该依赖低层模块，两者都应该依赖抽象
- 抽象不应该依赖细节，细节应该依赖抽象

**在本框架中的应用**：
```java
// 框架层定义 SPI 接口（抽象）
public interface CacheProvider {
    void set(String key, Object value);
}

// 基础设施层实现（细节）
@Service
public class RedisCacheService implements CacheProvider { }

// 框架层使用（依赖抽象）
@Service
public class MyService {
    private final CacheProvider cache;  // 不依赖具体实现
}
```

**优势**：
- 可以替换底层实现（如 Redis 换成 Memcached）而不影响上层代码
- 便于单元测试，可以 Mock 依赖

### 5. SPI 扩展机制

**理论来源**：Java SPI (Service Provider Interface) 机制

**核心思想**：框架定义接口，业务方提供实现，框架通过服务发现机制加载实现。

**工作流程**：
```
1. 框架定义接口
   public interface DomainContributor { ... }

2. 业务方实现接口
   @Component
   public class MyDomainContributor implements DomainContributor { ... }

3. 框架自动发现并加载
   @Autowired
   private List List<DomainContributor> contributors;  // 自动注入所有实现

4. 框架调用业务实现
   for (DomainContributor c : contributors) {
       c.contribute();
   }
```

**优势**：
- **开闭原则**：对扩展开放，对修改关闭
- **插件化**：新增功能只需添加实现类
- **解耦**：框架与业务实现完全解耦

## 适用场景

### 推荐使用

- **企业知识助手**：基于私有知识库的智能问答
- **业务流程自动化**：复杂业务场景的自动化处理
- **多 Agent 协作系统**：需要多个专业 Agent 协同工作
- **AI 中台建设**：企业级 AI 能力统一接入和管理

### 不推荐使用

- **简单单 Agent 应用**：直接使用 AgentScope 更轻量
- **对延迟极度敏感**：多层封装有一定开销
- **资源受限环境**：框架较重，需要一定资源
